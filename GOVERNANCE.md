# SplitMind 社区治理模型

## 概述

SplitMind 采用 **BDFL（Benevolent Dictator For Life）+ 核心维护团队** 的治理模型，确保项目既能快速决策，又能充分吸纳社区意见。

## 治理结构

### 1. BDFL（项目创始人）

**职责**：
- 项目愿景和方向把控
- 最终决策权（在争议时）
- 维护团队任命

**当前 BDFL**：项目创始人

### 2. 核心维护团队（Core Maintainers）

**组成**：3-5 名活跃贡献者

**职责**：
- 日常代码审查和合并
- 版本发布管理
- 社区问题响应
- 新贡献者指导

**任命方式**：
- 由 BDFL 提名
- 社区讨论（1 周）
- 无重大反对即通过

### 3. 贡献者（Contributors）

**分级**：

#### 3.1 新手贡献者（New Contributor）
- 首次提交 PR
- 获得欢迎和指导
- 目标：完成第一个贡献

#### 3.2 活跃贡献者（Active Contributor）
- 合并 3+ PR
- 参与 Issue 讨论
- 帮助其他新手

#### 3.3 核心贡献者（Core Contributor）
- 合并 10+ PR
- 参与技术决策
- 代码审查权限

## 决策流程

### 日常决策（维护团队）

**范围**：
- Bug 修复
- 文档更新
- 小型功能改进
- 代码审查

**流程**：
1. 提交 PR
2. 至少 1 名维护者审查通过
3. 合并到主分支

### 重大决策（社区投票）

**范围**：
- 架构变更
- API 不兼容修改
- 新增大型功能
- 许可证变更
- 商业化策略调整

**流程**：
1. 提交 RFC（Request for Comments）
2. 社区讨论（至少 2 周）
3. 投票（维护团队 + 核心贡献者）
4. BDFL 最终确认

**投票规则**：
- 赞成票 > 2/3 通过
- 平票时 BDFL 决定

## 贡献者分级激励机制

### 新手 → 活跃

**激励措施**：
- 首次贡献徽章
- 社区感谢信
- 新手指导计划

### 活跃 → 核心

**激励措施**：
- 代码审查权限
- 决策投票权
- 社区展示（Contributors 页面）

### 核心 → 维护者

**激励措施**：
- 合并权限
- 版本发布权限
- 项目治理参与权

## 社区沟通渠道

### 异步沟通
- **GitHub Issues**：Bug 报告、功能请求
- **GitHub Discussions**：技术讨论、问答
- **邮件列表**：重要公告

### 同步沟通
- **Discord/微信群**：日常交流
- **月度社区会议**：语音/视频会议
- **年度开发者大会**：线下聚会

## 争议解决

### 技术争议
1. 在 Issue/PR 中讨论
2. 寻求更多社区意见
3. 维护团队投票决定
4. 必要时 BDFL 裁决

### 社区争议
1. 私下沟通解决
2. 寻求调解人
3. 维护团队介入
4. 严重情况：暂时或永久封禁

## 财务透明

### 收入来源
- SaaS 托管服务
- 企业定制开发
- 培训认证
- 捐赠和赞助

### 支出用途
- 基础设施费用
- 开发工具和服务
- 社区活动
- 贡献者奖励

### 透明报告
- 每季度发布财务报告
- 公开收入和支出明细
- 社区监督和建议

## 路线图决策

### 短期（3个月）
- 维护团队决定
- 社区反馈收集

### 中期（6-12个月）
- 社区讨论
- 维护团队规划
- BDFL 确认

### 长期（1年+）
- 社区愿景讨论
- BDFL 方向把控
- 定期回顾调整
