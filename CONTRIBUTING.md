参与 SplitMind 项目的所有贡献者都应遵循我们的 [行为准则](CODE_OF_CONDUCT.md)。请确保您在参与项目时尊重他人，保持专业和友善的态度。

5. **反馈贡献**
   - 报告 bug
   - 提出新功能建议
   - 提供使用反馈

## 开发环境设置

```bash
git clone https://github.com/Mby159/SplitMind.git
cd SplitMind
```

### 2. 创建虚拟环境

```bash
# 使用 venv
python -m venv venv

# 激活虚拟环境
# Windows
venv\Scripts\activate

source venv/bin/activate
```

### 3. 安装依赖

```bash

pip install -e "[all,dev]"
```

### 4. 运行测试

```bash
pytest tests/ -v --cov=splitmind
```

## 代码风格

SplitMind 项目使用以下代码风格工具：

- **Black**：代码格式化
- **Ruff**：代码检查
- **MyPy**：类型检查

### 运行代码风格检查

```bash
# 检查代码风格
ruff check splitmind

# 格式化代码
black splitmind

# 类型检查
mypy splitmind
```

### 代码风格指南

1. **命名约定**
   - 类名：驼峰命名法（如 `PrivacyHandler`）
   - 函数名：蛇形命名法（如 `detect_sensitive_info`）
   - 变量名：蛇形命名法（如 `redacted_text`）
   - 常量名：全大写蛇形命名法（如 `MAX_TOKEN_LIMIT`）

2. **代码结构**
   - 每行不超过 100 个字符
   - 函数长度尽量控制在 50 行以内
   - 类长度尽量控制在 300 行以内
   - 模块长度尽量控制在 500 行以内

3. **注释**
   - 为公共 API 添加文档字符串
   - 为复杂逻辑添加注释
   - 避免冗余注释

4. **导入**
   - 按标准库、第三方库、本地库的顺序导入
   - 使用相对导入（如 `from .base import BaseProvider`）

## 提交规范

### 提交消息格式

```
<type>(<scope>): <subject>

<body>

<footer>
```

### 类型说明

- **feat**：新功能
- **fix**：bug 修复
- **docs**：文档更改
- **style**：代码风格更改（不影响功能）
- **refactor**：代码重构（不添加功能或修复 bug）
- **perf**：性能优化
- **test**：测试相关
- **chore**：构建过程或辅助工具的更改
- **ci**：CI 配置更改

### 示例

```
feat(providers): add support for Google Gemini provider

Adds a new provider implementation for Google Gemini models, including support for chat and completion capabilities.

Closes #123
```

## Pull Request 流程

1. **Fork 仓库**
   - 在 GitHub 上 fork 项目仓库

2. **创建分支**
   - 从 `main` 分支创建新的特性分支
   - 分支命名格式：`feature/feature-name` 或 `fix/bug-fix`

3. **进行更改**
   - 实现功能或修复 bug
   - 编写测试
   - 更新文档

4. **运行测试**
   - 确保所有测试通过
   - 运行代码风格检查

5. **提交更改**
   - 编写清晰的提交消息
   - 推送到您的 fork

6. **创建 PR**
   - 从您的分支创建 Pull Request 到 `main` 分支
   - 填写 PR 模板
   - 关联相关 Issue

7. **代码审查**
   - 回应审查意见
   - 进行必要的更改

8. **合并**
   - 一旦 PR 被批准，维护者将合并您的更改

## 测试指南

### 编写测试

1. **单元测试**
   - 测试单个功能或方法
   - 位于 `tests/` 目录
   - 文件名格式：`test_*.py`

2. **集成测试**
   - 测试多个组件的交互
   - 位于 `tests/integration/` 目录

3. **测试最佳实践**
   - 测试应该是独立的
   - 测试应该覆盖正常和异常情况
   - 测试应该有明确的断言
   - 测试应该有描述性的名称

### 运行测试

```bash
# 运行所有测试
pytest tests/ -v

# 运行特定测试文件
pytest tests/test_core.py -v

# 运行特定测试函数
pytest tests/test_core.py::test_privacy_handler -v

# 生成覆盖率报告
pytest tests/ --cov=splitmind --cov-report=html
```

- 当更改 API 时，更新相关文档

## 代码审查

### 审查原则

- **代码质量**：代码应该清晰、简洁、可维护
- **功能正确性**：代码应该实现预期功能
- **性能**：代码应该高效
- **安全性**：代码应该安全
- **测试覆盖**：代码应该有适当的测试覆盖

### 审查流程

1. **初始审查**：检查代码结构和逻辑
2. **详细审查**：检查具体实现细节
3. **测试审查**：检查测试覆盖
4. **文档审查**：检查文档更新
5. **批准**：如果所有检查通过，批准 PR

### 审查反馈

- 提供具体的反馈
- 解释为什么需要更改
- 提供改进建议
- 保持建设性和尊重

## 发布流程

### 版本号规范

SplitMind 使用语义化版本（Semantic Versioning）：

- **MAJOR**：不兼容的 API 变更
- **MINOR**：向后兼容的功能新增
- **PATCH**：向后兼容的问题修复

### 发布步骤

1. **更新 CHANGELOG.md**：记录版本变更
2. **更新版本号**：在 `pyproject.toml` 中更新版本号
3. **运行测试**：确保所有测试通过
4. **构建包**：构建 Python 包
5. **发布包**：发布到 PyPI
6. **创建标签**：创建版本标签
7. **推送标签**：推送标签到 GitHub

## 成为维护者

### 维护者职责

- 审查 Pull Request

- 维护代码质量
- 发布版本
- 指导新贡献者

### 如何成为维护者

1. **持续贡献**：定期为项目做出高质量的贡献
2. **参与讨论**：积极参与项目讨论和决策
3. **展示能力**：展示对项目的深度理解和技术能力
4. **获得认可**：得到现有维护者的认可
