# 为什么创建 SplitMind？隐私优先的 AI 任务编排新范式

> 在 AI 时代，隐私保护不应该是一种妥协，而应该是一种基础设施。

## 问题的起源

随着大型语言模型（LLM）的快速发展，我们越来越依赖 AI 来处理各种任务。但在这个过程中，一个根本性的问题被忽视了：**我们的敏感数据正在无意识地暴露给 AI 提供商**。

想象一下这些场景：
- 你让 AI 分析一份包含客户个人信息的报告
- 你让 AI 处理包含公司财务数据的文档
- 你让 AI 审查包含员工隐私信息的合同

在这些场景中，你的敏感数据被完整地发送给了 OpenAI、Anthropic 或其他 AI 提供商。即使他们承诺保护数据，但数据一旦离开你的控制范围，风险就产生了。

## 现有的解决方案及其局限

### 1. 本地部署模型
**方案**：在自己的服务器上部署开源模型如 Llama、Mistral。

**局限**：
- 需要昂贵的硬件资源
- 模型能力通常不如商业模型
- 维护和更新成本高

### 2. 数据脱敏后发送
**方案**：手动删除敏感信息后再发送给 AI。

**局限**：
- 耗时且容易遗漏
- 破坏数据的上下文和语义
- 难以处理复杂的敏感信息（如隐含的关联信息）

### 3. 使用企业版 API
**方案**：使用提供商的企业版，承诺不训练用户数据。

**局限**：
- 数据仍然需要离开你的环境
- 依赖提供商的承诺
- 成本较高

## SplitMind 的解决方案

SplitMind 采用了一种全新的思路：**任务拆分 + 隐私保护 + 多提供商编排**。

### 核心思想

1. **任务拆分**：将复杂任务拆分为多个独立的子任务
2. **隐私保护**：在本地对敏感信息进行检测和脱敏
3. **分布式执行**：将脱敏后的子任务分配给不同的 AI 提供商
4. **结果聚合**：在本地聚合结果，并还原敏感信息

### 工作流程

```
原始任务（包含敏感信息）
    ↓
[本地] 隐私保护处理（检测并脱敏敏感信息）
    ↓
[本地] 任务拆分为多个子任务
    ↓
子任务 A → OpenAI
子任务 B → Anthropic Claude
子任务 C → Kimi
子任务 D → 本地模型
    ↓
[本地] 结果聚合
    ↓
[本地] 敏感信息还原
    ↓
最终结果
```

### 关键优势

1. **隐私保护**：敏感信息在本地处理，不会暴露给任何 AI 提供商
2. **能力最大化**：可以使用最强的商业模型处理脱敏后的任务
3. **供应商独立性**：不依赖单一 AI 提供商，避免 vendor lock-in
4. **成本优化**：可以根据任务复杂度选择不同的模型，优化成本

## 技术实现亮点

### 1. 智能任务拆分
SplitMind 不是简单地将文本分段，而是理解任务的语义结构：
- 分析任务类型（分析、总结、翻译、创意等）
- 识别任务之间的依赖关系
- 根据隐私风险选择拆分策略

### 2. 上下文感知的隐私保护
传统的正则表达式匹配容易误报或漏报。SplitMind 采用：
- 多模式匹配（正则 + 语义分析）
- 上下文验证（如区分电话号码和快递单号）
- 风险分级（low/medium/high/critical）

### 3. 灵活的结果聚合
不同的任务需要不同的聚合策略：
- **顺序组合**：适用于需要保持顺序的任务
- **并行合并**：适用于多角度分析的任务
- **投票选择**：适用于需要高质量答案的任务
- **智能合并**：自动去重和补充

## 使用场景示例

### 场景 1：企业合同审查
```python
# 原始合同包含：公司名称、金额、员工信息、商业条款
result = await engine.execute(
    task="审查这份合同的潜在风险",
    context=contract_text,
    execution_mode="hybrid"  # 隐私保护在本地，审查使用在线模型
)
# 敏感信息被脱敏后发送，结果中自动还原
```

### 场景 2：医疗报告分析
```python
# 原始报告包含：患者姓名、身份证号、病史、诊断结果
result = await engine.execute(
    task="分析这份医疗报告的关键指标",
    context=medical_report,
    execution_mode="local_only"  # 最高隐私保护
)
# 所有处理都在本地完成，敏感信息不会离开本地环境
```

### 场景 3：市场研究报告生成
```python
# 需要整合多个数据源，部分包含敏感信息

```
